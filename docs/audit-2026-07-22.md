# HazardOS Codebase Audit — 22 July 2026

Six parallel audits: security, database, performance, accessibility/UX, code quality, best practices.
~90 findings. This document consolidates and ranks them.

**Verification status is marked on every P0 item.** Findings marked *verified live* were reproduced
against the running database or by executing the exploit; everything else is agent-reported with a
file:line citation and has **not** been independently confirmed. Verify before fixing.

---

## Two systemic patterns

Almost every serious finding is an instance of one of these.

### Pattern A — Silent success

Code reports success while doing nothing. Every instance fails in the direction that hides the failure.

| Reported | Reality |
|---|---|
| Cron run `ok` | No reminder ever sent (`lib/services/reminder-sender.ts:1`) |
| Stripe `200 {received:true}` | Subscription never activates (`lib/services/stripe-service.ts:263`) |
| SMS "sent" | Delivery status never recorded (`lib/services/sms-service.ts:845`) |
| "Saved locally, will submit when online" | Never submits (`mobile-survey-wizard.tsx:492`) |
| Review screen shows N photos | Bytes discarded on reload (`lib/stores/photo-queue-store.ts:223`) |
| Wizard advances a step | Validation failed, logged to `debug`, ignored (`mobile-survey-wizard.tsx:357`) |
| Variance recalculated | RPC errored, fell back silently (`lib/services/job-variance-service.ts:12`) |
| "No hardcoded secrets detected" | Regex can never match (`.github/workflows/security.yml:103`) |
| Review queue empty | Query failed, error discarded (`jobs/review-queue/page.tsx:23`) |

Root cause in the DB cases: **background/webhook code uses the cookie-based anon client where there is
no session.** RLS then returns zero rows or refuses the write with no error. The fix is a shared
"assert the write landed" helper plus `createAdminClient()` in sessionless contexts — not nine
individual patches.

### Pattern B — Authorization enforced in application code, not in the database

The API gates with `allowedRoles`; the database mostly gates on `organization_id` alone. Anyone who
talks to PostgREST directly with the anon key — which ships in the browser bundle — bypasses the API
entirely.

- 68 tables have no role predicate on writes (`viewer` is not read-only at the database)
- 85 write endpoints omit `allowedRoles` altogether
- `tenant_invitations.role` has no CHECK constraint; the `z.enum` guard is app-side only
- `profiles` UPDATE has no `WITH CHECK` (see P0-1)

## P0 — Fix immediately

### P0-1. Privilege escalation via `profiles` — **verified live, exploit reproduced**

`profile_own_update` and `profiles_update_own` are both `FOR UPDATE USING (id = auth.uid())` with **no
`WITH CHECK`**. Postgres reuses `USING` as the check, so only `id` is pinned. `role`,
`organization_id` and `is_platform_user` are freely writable by the row's owner.

Reproduced end-to-end with a `viewer` account and the public anon key:

```
BEFORE : {"role":"viewer",         "organization_id":"61033d27…", "is_platform_user":false}
AFTER  : {"role":"platform_owner", "organization_id":"c8a2cb0f…", "is_platform_user":true}
ESCALATED: true    MOVED INTO VICTIM ORG: true
```

No trigger guards these columns. This defeats the entire authorization model: `get_user_role()` then
returns `platform_owner`, satisfying every `PLATFORM_ONLY` policy and every `allowedRoles` check.
Every other control in the system — including the SEC22 proposal fix shipped the same day — is
downstream of a value the attacker chooses.

**Fix:** `WITH CHECK` pinning the three columns to their current values, plus a `BEFORE UPDATE`
trigger as defence in depth. Note there are *duplicate* policy pairs on `profiles` and policies OR
together — a partial fix has no effect. Both copies must be corrected or dropped.

### P0-2. `anon` can execute RLS-bypassing RPCs — **verified live**

Confirmed via `has_function_privilege`: `get_top_slow_queries`, `reset_rate_limit`,
`check_and_increment_rate_limit`, `recalc_customer_stats`, `recalc_company_stats` are all
`SECURITY DEFINER` with `anon EXECUTE = true` and no tenant check.

Disclosure proven: `get_top_slow_queries` returns live `pg_stat_statements` rows including
PostgREST-generated queries against `public.organizations` — schema and query structure for every
tenant, readable unauthenticated at `POST /rest/v1/rpc/get_top_slow_queries`.
`reset_rate_limit` performs an unconditional `UPDATE api_keys SET rate_limit_count = 0`.

**Fix:** `REVOKE EXECUTE … FROM anon` on all five; add tenant checks to the `recalc_*` pair.

### P0-3. `calculate_completion_variance_by_job` — cross-tenant write — **verified live**

`SECURITY DEFINER`, `GRANT EXECUTE TO authenticated`, **no `SET search_path`**, no organization check.
A technician at org A can pass org B's job UUID and silently rewrite org B's cost variance and margin.

**This was made worse by `20260721000004_populate_actual_total_and_margin.sql`, written during this
session.** The vulnerability pre-existed, but that migration recreated the function, carried
`SECURITY DEFINER` and the `authenticated` grant forward, and added neither `SET search_path` nor a
tenant check — while the proposal RPCs written hours later got `search_path=public` correctly.

**Fix:** add `SET search_path = public`, add an `organization_id` check against the caller, or make it
`SECURITY INVOKER`.

## P3 — Performance

1. Every authenticated request: `auth.getUser()` twice (proxy + handler) + `profiles` select + Redis — 4 sequential hops before any logic.
2. `app/(dashboard)/layout.tsx` is `'use client'` and gates children on 3 serial client round-trips, discarding all server rendering.
3. Unbounded list endpoints: `/site-surveys` (16 cols + 4 embeds), `/api/work-orders`, `/api/lab-reports` (`select('*')` + 5 embeds). PostgREST silently truncates, making `chain_total` wrong.
4. Pipeline kanban: 1,000 × ~55-column rows, no virtualization.
5. Missing composites: `activity_log(organization_id, created_at DESC)` (highest-write table; `audit_log` got this and `activity_log` was missed), `notifications(user_id, created_at DESC)` (polled every 30s with exact count).
6. `POST /api/jobs` blocks the response on an OAuth refresh + Google/Outlook write — 2–4s.
7. 116 FKs unindexed, including `organization_id` on 14 tables — `profiles.organization_id` is evaluated on effectively every request.
8. Dashboard charts bypass TanStack Query (raw `useEffect` + `fetch`), refetching on every mount.

## P4 — Accessibility

1. Pipeline kanban is pointer-only — no `KeyboardSensor`; no alternative stage-change control. The CRM's headline feature is unusable by keyboard (WCAG 2.1.1).
2. `<Label>` used without `htmlFor` across ~78 files — most data entry announces as "edit text, blank". **`components/ui/form-field.tsx` already does this correctly and is used in 5 files** — an adoption problem, not a design problem.
3. Table rows navigate on click with no keyboard path and no `<a href>` — jobs/invoices lists can't be keyboard-navigated or link-copied.
4. Cards/rows as `<div onClick>` with no role/tabIndex — the hazards step has no other route in.
5. 18 dashboard pages have no `<h1>`; several have no headings at all.
6. Icon-only back buttons unnamed on 3 CRM detail pages (only 4 such buttons in the codebase — cheap fix).
7. API fetch failures render as empty states, not errors — users are told a customer has no contacts when the call failed.
8. No unsaved-changes guard on any desktop form.

## P5 — Sustainability / operations

1. No timeouts on any QuickBooks/HubSpot/Mailchimp/Google/Outlook fetch — a partner's slowdown becomes your 504.
2. Outbound webhook retries compute `next_retry_at`; nothing ever reads it.
3. `pnpm.overrides` is dead config (repo uses npm); `security.yml` reads that block instead of the live `overrides`. Installed `minimatch@3.1.5`, `ajv@6.14.0` are below their supposed pins.
4. Service worker caches tenant REST data and storage objects keyed on URL only; sign-out doesn't purge — previous user's records and job photos remain on a shared field tablet.
5. Client-side Sentry only initializes under `--webpack`; disappears silently if that flag is dropped. `sentry.edge.config.ts` lacks the scrubbing its server sibling has.
6. Sentry tunnel `/monitoring` and `/api/errors/report` aren't in `PUBLIC_ROUTES` — every browser error on login/signup/portal is dropped.
7. `.env.example` documents 12 vars; code reads at least 15 more, including `CRON_SECRET` (without it every cron 500s).
8. Supabase project ref hardcoded in `next.config.mjs:11`.
9. Two full PDF stacks (`@react-pdf` + `jspdf`); ~980 lines of proposal PDF code is entirely dead. `experimental.optimizeCss` enabled without `beasties`/`critters` installed.
10. Three legacy/CRM route pairs were collapsed to redirects; **jobs was not** — `/jobs` and `/crm/jobs` are both live full implementations with different filters, pagination and status configs.

## Recommended sequence

1. **P0 as one hardening migration**, each verified by re-running its exploit script.
2. **Pattern A as one change** — `createAdminClient()` in sessionless contexts + a shared assert-the-write-landed helper. Fixes P1-3, P1-4, P1-14 and prevents recurrence.
3. **Pattern B** — role predicates on write policies, `CHECK` on `tenant_invitations.role`, `allowedRoles` on the 85 endpoints.
4. **P2-1 (type-check the tests)** before the other P2 work — it converts 115 route tests into real contract tests and would have caught P2-3 and P2-4.
5. P1 data-loss items (photos, offline submit, cascade deletes).
6. P3/P4/P5 as normal backlog.
